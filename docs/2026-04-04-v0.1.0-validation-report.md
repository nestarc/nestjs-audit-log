# 0.1.0 Validation Report

Date: 2026-04-04
Repository: `@nestarc/audit-log`

## Review Team

- Architecture lead: NestJS module/API/runtime flow review
- Security lead: audit integrity, tenant isolation, append-only guarantees
- Release QA lead: test coverage, packaging, reproducibility, docs

## Verdict

- Status: hold external release
- Summary: the package builds and its current unit/E2E suite passes, but two release-blocking gaps remain: audit writes are not atomic with the business mutation, and clean-environment E2E/bootstrap is not reproducible from the published scripts/docs.

## Evidence Collected

- `npm test`: passed, 8/8 suites and 44/44 tests
- `npm run build`: passed
- `npm run test:cov`: passed, overall statement coverage 62.94%
- `npm run prisma:test:generate`: passed
- `npm run prisma:test:push`: failed with `Schema engine error`
- `npm run test:e2e`: passed only after manually creating the test tables in PostgreSQL

## Findings

### 1. Critical: audit logging is not transactional with the original write

- Evidence: [`src/prisma/audit-extension.ts`](../src/prisma/audit-extension.ts) lines 104-116, 129-144, 157-171, 184-205, 216-245, 260-274
- The extension executes the Prisma mutation first and inserts the audit row afterward. If the audit insert fails, the business row is already committed but the caller still receives an error. That creates the worst possible failure mode for an audit library: user-visible failure, durable data change, and missing audit trail.
- Risk: data/audit divergence, duplicate retries, compliance failure, hard-to-explain incident timelines.
- Recommendation: execute the business mutation and audit insert inside one database transaction boundary, or explicitly design the package around an outbox/event pipeline with documented durability semantics.

### 2. High: projected Prisma results can produce incomplete or malformed audit rows

- Evidence: [`src/prisma/audit-extension.ts`](../src/prisma/audit-extension.ts) lines 104-114, 129-141, 184-203
- `targetId` and change computation are derived from the returned `result`. When callers use `select` or `include` to omit `id` or other fields, the audit entry can lose the target identifier and the diff will only reflect projected fields instead of the actual row mutation.
- Risk: audit records become partial, misleading, or impossible to correlate to the changed entity.
- Recommendation: fetch a canonical record for audit purposes independent of the caller projection, or require/validate presence of the primary key before writing the audit row.

### 3. High: release bootstrap is not reproducible from a clean environment

- Evidence: [`package.json`](../package.json) lines 11-18, [`README.md`](../README.md) lines 1-2
- The documented package surface is effectively empty, and the provided `prisma:test:push` script does not currently provision the E2E schema in this environment. I had to create `users` and `audit_logs` manually before `npm run test:e2e` would pass.
- Risk: contributors and CI cannot reliably reproduce the advertised test path; first-time adopters do not have install/setup instructions for the required schema objects.
- Recommendation: fix the schema provisioning path, document exact setup steps, and add a one-command verified E2E bootstrap flow.

### 4. High: the append-only guarantee exists only in test code, not in the package contract

- Evidence: [`test/e2e/audit-log.e2e-spec.ts`](../test/e2e/audit-log.e2e-spec.ts) lines 31-43, [`README.md`](../README.md) lines 1-2
- The package description claims append-only PostgreSQL storage, but the only place that creates the no-update/no-delete rules is the E2E test harness. The library does not ship migrations, SQL, or operational documentation that establishes this guarantee for consumers.
- Risk: consumers can publish `0.1.0` believing append-only protection exists when production deployments actually allow direct mutation/deletion of audit rows.
- Recommendation: ship migration SQL or a documented installation helper, and make the guarantee conditional in docs until it is actually provisioned by the package.

### 5. Medium: tenant integration is fail-open and package metadata does not declare the optional peer correctly

- Evidence: [`src/utils/tenant.ts`](../src/utils/tenant.ts) lines 1-7, [`src/services/audit.service.ts`](../src/services/audit.service.ts) lines 47-53, [`package.json`](../package.json) lines 19-29
- `getTenantId()` swallows all errors and returns `null`, after which `query()` simply omits the tenant predicate. In addition, `peerDependenciesMeta` marks `@nestarc/tenancy` as optional without actually declaring it in `peerDependencies`.
- Risk: tenant filtering can silently disappear when tenancy wiring is broken, and package managers do not get an accurate dependency contract.
- Recommendation: declare the optional peer explicitly, distinguish "tenancy package absent" from "tenant context missing/error", and consider a configurable fail-closed mode for multi-tenant deployments.

### 6. Medium: change detection is shallow and under-tested for real Prisma payloads

- Evidence: [`src/prisma/diff.ts`](../src/prisma/diff.ts) lines 35-49, coverage from `npm run test:cov`
- `computeUpdateChanges()` uses reference inequality. JSON fields, arrays, and other structured values will generate noisy diffs even when semantically unchanged, while projection-based results reduce fidelity further. Coverage is especially low in [`src/prisma/audit-extension.ts`](../src/prisma/audit-extension.ts).
- Risk: large or inaccurate diffs in production logs, especially on models with JSON columns.
- Recommendation: add targeted tests for JSON/date/select/include/upsert/createMany/updateMany cases and switch to a stable comparison strategy for structured values.

## Strengths Observed

- The current unit suite is fast and stable.
- Core NestJS wiring (`forRoot`, `forRootAsync`, middleware, interceptor, service query API) is covered at a basic level.
- Raw SQL usage is parameterized in the audited paths that were reviewed.
- Manual logging, wildcard querying, redaction, and basic CUD tracking all work in the tested happy paths.

## Release Gate

- Must fix before publish: transactional audit consistency
- Must fix before publish: reproducible schema/bootstrap path plus real documentation
- Should fix before broader adoption: projection-safe auditing and tenant fail-open behavior
- Should add before 0.2.0: stronger coverage around Prisma edge cases and append-only installation automation
