# 0.1.0 Validation Report

Date: 2026-04-04
Repository: `@nestarc/audit-log`

## Review Team

- Architecture/API lead: NestJS module wiring, public API, Prisma extension behavior
- Security/integrity lead: audit record correlation, append-only path, tenant/actor propagation
- QA lead: unit/E2E reproducibility, regression gaps, release confidence
- Release/docs lead: package surface, install flow, README accuracy, publish readiness

## Verdict

- Status: hold external release
- Summary: core behavior is working, but 0.1.0 still has contract gaps that are too large for a public first release. The package builds, packs, unit tests pass, and E2E behavior passes after manual schema provisioning. However, the library currently assumes tracked models expose an `id` field, manual logging is not transaction-safe with caller-owned business transactions, and the advertised clean-environment E2E/bootstrap path is not reproducible.

## Findings

### 1. High: tracked model support silently assumes a scalar `id` field

- Evidence: [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L120), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L125), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L179), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L215), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L247), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L354)
- The extension derives `targetId` from `(result as any).id` and fetches canonical rows with `where: { id: ... }`. Models that use another primary key name, composite keys, or projections that omit `id` will generate incomplete audit rows or skip canonical fetches entirely.
- Consumer impact: a library marketed as generic Prisma model tracking will silently degrade on valid Prisma schemas outside the narrow `id` convention.
- Recommendation: detect primary key metadata instead of hard-coding `id`, or document and enforce the `id`-only limitation before publish.

### 2. High: `AuditService.log()` is not transaction-safe with caller-owned business transactions

- Evidence: [`src/services/audit.service.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/services/audit.service.ts#L21), [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L8), [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L11)
- Inference from source: manual logging always writes through the base Prisma client's `$executeRaw` and offers no way to bind to an existing caller transaction. If an application emits a manual audit event inside a broader business flow that later rolls back, the audit row can still commit independently.
- Consumer impact: the README's broad "same database transaction" claim does not hold for the manual logging API, which is precisely the path consumers will use for business events outside the automatic Prisma extension.
- Recommendation: either add a transaction-aware manual logging API that accepts a Prisma transaction client, or narrow the documentation so the transactional guarantee is explicitly limited to the automatic extension path.

### 3. High: the README does not show the required runtime wiring for automatic auditing

- Evidence: [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L40), [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L45), [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L61), [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L62)
- The guide creates an extended Prisma client in step 3, but step 4 registers the module with `basePrisma` and never shows that application writes must run through the extended client. A consumer can follow the README closely and still end up with manual logging working while automatic CUD tracking never fires.
- Consumer impact: first adopters can get a false-positive setup, especially in Nest apps that keep injecting a non-extended `PrismaService`.
- Recommendation: show one complete Nest integration path where the app's Prisma provider is the extended client used for business writes, and explicitly separate "base client for audit storage" from "extended client for audited mutations" if both are needed.

### 4. High: the advertised clean-environment E2E/bootstrap path is not reproducible

- Evidence: [`package.json`](/Users/ksy/Documents/GitHub/nestjs-audit-log/package.json#L15), [`package.json`](/Users/ksy/Documents/GitHub/nestjs-audit-log/package.json#L20), [`README.md`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L149), [`test/e2e/audit-log.e2e-spec.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/test/e2e/audit-log.e2e-spec.ts#L31)
- On 2026-04-04, `npm run prisma:test:push` failed with `Schema engine error`. Separately, `npm run test:e2e` failed on a clean database because the suite assumes `audit_logs` already exists before it creates append-only rules. The same E2E suite passed only after manually provisioning the tables in PostgreSQL.
- Consumer impact: contributors and CI cannot rely on the documented one-command E2E path, and the project's own release readiness signal is weaker than the README suggests.
- Recommendation: make schema provisioning deterministic, then keep `test:e2e:full` as a verified happy path in CI.

### 5. Medium: `update()` can lose canonical diffs when the updated unique field is part of `where`

- Evidence: [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L159), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L165), [`src/prisma/audit-extension.ts`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L172)
- The code re-fetches the post-update record with the original `args.where`. If that unique field itself changed, the second lookup misses and diff generation falls back to the projected `result`, which may omit fields that actually changed.
- Consumer impact: audit trails can become incomplete exactly on the kinds of identity-changing updates that are often sensitive.
- Recommendation: re-fetch using the canonical primary key captured before the update, not the pre-update `where` filter.

## Strengths Observed

- The package builds cleanly and `npm pack --dry-run` shows the intended publish payload, including the shipped SQL file.
- Unit coverage is broad enough to stabilize the core APIs: module registration, middleware, interceptor, diff helpers, service query path, and helper functions.
- Once the test schema exists, the current E2E suite passes for create, update, delete, `deleteMany`, redaction, manual logs, and query pagination/filtering.
- The public surface is compact and understandable for a 0.1.0 package.

## Residual Risks

- Inference from source: outer application transactions are not covered by tests, while the extension opens its own `client.$transaction(...)` inside each write hook. Add an explicit regression test for "caller transaction rolls back both business row and audit row" before broader adoption.
- There is no regression coverage for non-`id` primary keys, composite keys, or caller projections that omit `id`.
- `upsert`, `createMany`, and `updateMany` are implemented but not covered by the current E2E suite, and the real Nest HTTP request path is not exercised end to end.
- The append-only rules are configured in SQL, but there is no behavioral regression test that proves `UPDATE audit_logs` and `DELETE audit_logs` are actually blocked.
- The current validation did not exercise a real `@nestarc/tenancy` installation, so multi-tenant behavior remains only partially validated.

## Release Gate

- Must fix before publish: document and/or enforce the tracked-model primary-key constraint
- Must fix before publish: narrow or repair the manual logging transactional guarantee
- Must fix before publish: restore a reproducible `test:e2e:full` bootstrap path
- Should fix before broader adoption: provide a complete Nest integration example that uses the extended Prisma client for application writes
- Should fix before broader adoption: re-fetch canonical update state without relying on the pre-update `where`
- Should add before 0.2.0: transaction rollback regression tests, schema-variation tests, and request-path append-only assertions
