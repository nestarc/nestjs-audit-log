# 0.1.0 Validation Report

Date: 2026-04-04
Repository: `@nestarc/audit-log`

## Review Team

- Architecture/runtime lead: Prisma extension behavior, NestJS integration, transaction model
- Security/integrity lead: tenant isolation, append-only guarantees, audit-record correctness
- QA/release lead: build, packaging, CI signal, E2E reproducibility
- API/docs lead: public option surface, README accuracy, adoption ergonomics
- Primary integrator: command execution, result verification, final synthesis

## Verdict

- Status: hold external release
- Summary: the package now builds, packs, and passes both unit and E2E validation, and the shipped append-only SQL works when provisioned. However, `0.1.0` still has two release-blocking integrity issues: the automatic Prisma extension does not participate in caller-owned transactions, and multi-tenant query isolation fails open when tenancy context resolution breaks. There are also public-contract gaps around module options, decorator scope, and projection handling.

## Environment Note

- Initial `prisma db push` / E2E failures observed in the sandbox were environment-induced. Localhost TCP access is blocked in the sandbox, so those failures were not treated as product defects.
- After rerunning with local Docker and localhost access enabled, the documented E2E bootstrap path completed successfully.

## Findings

### 1. High: automatic audit writes do not join caller-owned Prisma transactions

- Evidence: [`src/prisma/audit-extension.ts#L125`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L125), [`src/prisma/audit-extension.ts#L166`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L166), [`src/prisma/audit-extension.ts#L213`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L213), [`src/prisma/audit-extension.ts#L251`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L251), [`src/prisma/audit-extension.ts#L303`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L303), [`src/prisma/audit-extension.ts#L330`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L330), [`src/prisma/audit-extension.ts#L357`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L357), [`README.md#L191`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L191)
- The extension ignores the intercepted `query(args)` path and re-executes every mutation inside a new `client.$transaction(...)`. That means it guarantees atomicity only for the re-issued business write plus the audit insert, not for the caller's surrounding transaction.
- Verified locally on 2026-04-04: an outer `prisma.$transaction(async (tx) => { await tx.user.create(...); throw ... })` left both the business row and the `User.created` audit row committed after the forced rollback.
- Consumer impact: applications that rely on interactive transactions can observe committed writes and committed audit rows even when their outer business transaction aborts.

### 2. High: multi-tenant isolation fails open when tenancy context lookup fails

- Evidence: [`src/utils/tenant.ts#L17`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/utils/tenant.ts#L17), [`src/utils/tenant.ts#L21`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/utils/tenant.ts#L21), [`src/services/audit.service.ts#L48`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/services/audit.service.ts#L48), [`src/services/audit.service.ts#L52`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/services/audit.service.ts#L52)
- Source-based inference: if `@nestarc/tenancy` is installed but `TenancyContext().getTenantId()` throws, `getTenantId()` logs a warning and returns `null`. `AuditService.query()` then omits the tenant predicate entirely, and writes proceed with `tenant_id = null`.
- Consumer impact: in a multi-tenant deployment, any tenancy-context failure can widen audit queries across tenants and weaken write-side partitioning.

### 3. Medium: the module option surface is misleading and invites silent misconfiguration

- Evidence: [`src/interfaces/audit-log-options.interface.ts#L4`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/interfaces/audit-log-options.interface.ts#L4), [`src/middleware/audit-actor.middleware.ts#L13`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/middleware/audit-actor.middleware.ts#L13), [`src/services/audit.service.ts#L21`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/services/audit.service.ts#L21), [`README.md#L58`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L58), [`README.md#L103`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L103)
- `AuditLogModuleOptions` exposes `trackedModels`, `ignoredModels`, and `sensitiveFields`, and the README repeats those fields inside `AuditLogModule.forRootAsync(...)`. At runtime, the module itself only consumes `actorExtractor` and `prisma`; the automatic tracking configuration actually lives in `createAuditExtension(...)`.
- Consumer impact: users can change module config and reasonably believe tracking or redaction changed, while the Prisma extension keeps running with its old settings.

### 4. Medium: class-level `@NoAudit()` and `@AuditAction()` are ignored

- Evidence: [`src/decorators/no-audit.decorator.ts#L4`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/decorators/no-audit.decorator.ts#L4), [`src/decorators/audit-action.decorator.ts#L4`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/decorators/audit-action.decorator.ts#L4), [`src/interceptors/audit.interceptor.ts#L17`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/interceptors/audit.interceptor.ts#L17), [`README.md#L167`](/Users/ksy/Documents/GitHub/nestjs-audit-log/README.md#L167)
- The decorators are general Nest metadata decorators, but the interceptor only reads `context.getHandler()` and never checks `context.getClass()`.
- Consumer impact: controller-level audit policy declarations are silently ignored, which is easy to miss in review and hard to detect without explicit tests.

### 5. Medium: `create` / `upsert` are not actually projection-safe when the primary key is omitted

- Evidence: [`src/prisma/audit-extension.ts#L128`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L128), [`src/prisma/audit-extension.ts#L133`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L133), [`src/prisma/audit-extension.ts#L254`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L254), [`src/prisma/audit-extension.ts#L262`](/Users/ksy/Documents/GitHub/nestjs-audit-log/src/prisma/audit-extension.ts#L262)
- The canonical re-fetch path depends on `result[pkField]`. When a valid Prisma `select` omits the primary key, `targetId` becomes `null` and the fallback audit payload only contains the projected fields.
- Verified locally on 2026-04-04: `prisma.user.create({ ..., select: { name: true } })` produced `{"action":"User.created","targetId":null,"changes":{"name":{"after":"Projection User"}}}`.
- Consumer impact: audit trails can silently lose identifiers and non-projected fields on successful writes.

## Strengths Observed

- The package builds cleanly and the dry-run publish artifact contains the expected runtime files, including the bundled SQL schema.
- The documented one-command E2E path is reproducible once the environment has access to local Docker and localhost networking.
- `AuditService.log()` already supports an explicit transaction client parameter, and the README now documents that contract correctly.
- The shipped append-only SQL rules do work when applied; direct `UPDATE` and `DELETE` attempts against `audit_logs` were prevented.

## Test and Release Risks

- CI only varies Node.js versions; it does not validate the advertised peer dependency matrix for Nest 10/11 and Prisma 5/6. See [`ci.yml#L13`](/Users/ksy/Documents/GitHub/nestjs-audit-log/.github/workflows/ci.yml#L13).
- The E2E suite boots a Nest application but mostly exercises the library through `AuditContext.run(...)` and direct service calls, so the real HTTP middleware + interceptor path is still lightly covered.
- `upsert`, `createMany`, and `updateMany` are implemented but not exercised by the current E2E suite.

## Release Gate

- Must fix before external release: make automatic auditing participate in caller-owned transactions, or narrow the public contract so the current behavior is explicit and enforced
- Must fix before external release: fail closed when tenancy context is required but unavailable, or make cross-tenant query widening impossible
- Should fix before broader adoption: remove or wire through the unused module options
- Should fix before broader adoption: honor class-level decorator metadata
- Should fix before broader adoption: preserve audit correctness when Prisma projections omit the primary key
- Should add before `0.2.0`: regression coverage for outer transaction rollback, HTTP request-path behavior, batch/upsert paths, and append-only enforcement
